## 2. 项目背景

用户在日常生活和工作中，经常需要快速记录一些轻量级待办事项，例如购物清单、临时任务、学习计划、工作提醒等。

本项目计划开发一个简单、易用的移动端 Todo List 应用，帮助用户在手机上快速创建、查看、完成和删除待办任务。

项目不做复杂的账号体系、任务分类、提醒、协作等功能，仅聚焦于最基础的待办事项管理逻辑，确保功能完整、交互清晰、实现成本可控。

---

## 3. 产品目标

### 3.1 核心目标

- 用户可以在手机端查看待办任务列表。
- 用户可以新增一条待办任务。
- 用户可以将任务标记为已完成或未完成。
- 用户可以删除不需要的任务。
- 用户可以按任务状态筛选列表，包括全部、未完成、已完成。
- 任务数据从后端接口获取，不依赖纯前端本地静态数据。

### 3.2 非目标范围

以下功能不包含在 V1.0 范围内：

- 用户注册、登录、退出登录。
- 多用户数据隔离。
- 任务截止时间。
- 任务提醒。
- 任务优先级。
- 任务分类、标签。
- 子任务。
- 任务搜索。
- 多端同步状态展示。
- 桌面端专门适配。

## 4. 用户角色

### 4.1 普通用户

不需要登录，进入页面后即可使用 Todo List 功能。

用户主要操作包括：

- 查看任务列表。
- 新增任务。
- 完成任务。
- 取消完成任务。
- 删除任务。
- 切换任务筛选状态。

## 5. 使用场景

### 5.1 快速记录任务

用户想到一件需要完成的事情，打开页面，在输入框中输入任务标题并提交，系统创建一条新的未完成任务。

### 5.2 查看待办事项

用户打开页面后，系统从后端获取任务列表，并默认展示全部任务。

### 5.3 完成任务

用户完成某个事项后，点击任务前的勾选控件，将任务状态更新为已完成。

### 5.4 筛选任务

用户可以通过「全部 / 未完成 / 已完成」筛选不同状态的任务列表。

### 5.5 删除任务

用户可以删除不再需要展示的任务。

## 6. 功能需求

## 6.1 任务列表

### 功能描述

用户进入页面后，系统展示任务列表。任务数据从后端接口获取。

### 展示内容

每条任务展示以下信息：

| 字段     | 说明                     |
| -------- | ------------------------ |
| 任务标题 | 用户创建任务时输入的文本 |
| 完成状态 | 未完成 / 已完成          |
| 创建时间 | 任务创建时间             |

### 展示规则

- 默认展示全部任务。
- 未完成任务应有明显的未完成状态。
- 已完成任务应有明显的完成状态，例如勾选态、文字弱化或删除线。
- 当任务列表为空时，展示空状态提示。
- 当数据加载中时，展示加载状态。
- 当接口请求失败时，展示错误提示，并允许用户重新加载。

---

## 6.2 新增任务

### 功能描述

用户可以创建新的待办任务。

### 操作流程

1. 用户在输入框中输入任务标题。
2. 用户点击新增按钮，或通过键盘提交。
3. 前端校验任务标题。
4. 校验通过后，请求后端创建任务。
5. 创建成功后，任务出现在列表中。
6. 新任务默认状态为未完成。

### 校验规则

| 规则     | 说明                   |
| -------- | ---------------------- |
| 必填     | 任务标题不能为空       |
| 去除空格 | 提交时去除标题前后空格 |
| 长度限制 | 建议最多 50 个字符     |
| 重复任务 | V1.0 不限制重复标题    |

### 异常处理

- 标题为空时，不允许提交，并提示用户输入任务内容。
- 创建失败时，提示用户创建失败，可重新尝试。
- 提交过程中按钮应避免重复点击。

---

## 6.3 完成任务 / 取消完成

### 功能描述

用户可以切换任务的完成状态。

### 操作流程

1. 用户点击任务前的勾选控件。
2. 系统请求后端更新任务完成状态。
3. 更新成功后，前端同步展示最新状态。

### 状态规则

| 当前状态 | 操作后状态 |
| -------- | ---------- |
| 未完成   | 已完成     |
| 已完成   | 未完成     |

### 异常处理

- 更新失败时，恢复原状态，并提示用户操作失败。
- 更新过程中可临时禁用当前任务的状态切换，避免重复请求。

---

## 6.4 删除任务

### 功能描述

用户可以删除一条任务。

### 操作流程

1. 用户点击任务上的删除按钮。
2. 系统展示删除确认态。
3. 用户确认删除后，请求后端删除任务。
4. 删除成功后，该任务从列表中移除。

### 交互规则

- 删除操作提供轻量二次确认，建议使用底部确认弹层。
- 用户点击取消时，关闭确认态，不发起删除请求。
- 删除成功后，列表立即更新。

### 异常处理

- 删除失败时，任务仍保留在列表中。
- 系统提示用户删除失败，可稍后重试。

---

## 6.5 任务筛选

### 功能描述

用户可以按任务状态筛选列表。

### 筛选项

| 筛选项 | 展示内容         |
| ------ | ---------------- |
| 全部   | 展示所有任务     |
| 未完成 | 仅展示未完成任务 |
| 已完成 | 仅展示已完成任务 |

### 交互规则

- 默认选中「全部」。
- 当前选中的筛选项需要有明显高亮状态。
- 切换筛选项后，列表内容即时更新。
- 筛选可以在前端基于已获取数据完成。
- 如果切换后的列表为空，展示对应空状态。

## 8. 数据字段

任务对象建议包含以下字段：

| 字段      | 类型            | 说明         |
| --------- | --------------- | ------------ |
| id        | string / number | 任务唯一标识 |
| title     | string          | 任务标题     |
| completed | boolean         | 是否完成     |
| createdAt | string          | 创建时间     |

示例数据：

```json
{
  "id": 1,
  "title": "完成 Todo List PRD",
  "completed": false,
  "createdAt": "2026-06-15T10:00:00Z"
}
```


## 9. 接口说明

为便于前后端联调和工作流验证，V1.0 建议采用以下接口契约。实际实现可按项目技术栈调整，但能力边界保持一致。

| 方法   | 路径             | 能力         | 说明                     |
| ------ | ---------------- | ------------ | ------------------------ |
| GET    | `/api/todos`     | 获取任务列表 | 页面初始化时获取全部任务 |
| POST   | `/api/todos`     | 创建任务     | 新增一条未完成任务       |
| PATCH  | `/api/todos/:id` | 更新任务状态 | 修改任务为已完成或未完成 |
| DELETE | `/api/todos/:id` | 删除任务     | 删除指定任务             |

### 请求和响应建议

#### 获取任务列表

- 请求：`GET /api/todos`
- 响应：任务对象数组。

#### 创建任务

- 请求：`POST /api/todos`
- 请求体：

```json
{
  "title": "完成 Todo List 页面原型"
}
```

- 响应：创建后的任务对象，新任务默认 `completed` 为 `false`。

#### 更新任务状态

- 请求：`PATCH /api/todos/:id`
- 请求体：

```json
{
  "completed": true
}
```

- 响应：更新后的任务对象。

#### 删除任务

- 请求：`DELETE /api/todos/:id`
- 响应：删除成功状态，或返回被删除的任务标识。

### 数据交互要求

- 所有任务数据应从后端接口获取。
- 前端不应使用固定写死的静态任务数据作为正式数据源。
- 接口异常时，前端需要展示错误提示。
- 创建、更新、删除成功后，前端需要同步刷新或更新当前列表状态。

## 10. 状态说明

## 10.1 加载状态

### 触发场景

- 首次进入页面。
- 手动重新加载任务列表。

### 展示要求

- 显示加载提示。
- 加载过程中避免页面空白。

---

## 10.2 空状态

### 触发场景

- 全部任务为空。
- 未完成筛选下无任务。
- 已完成筛选下无任务。

### 文案建议

| 场景       | 文案           |
| ---------- | -------------- |
| 全部为空   | 暂无待办事项   |
| 未完成为空 | 暂无未完成任务 |
| 已完成为空 | 暂无已完成任务 |

---

## 10.3 错误状态

### 触发场景

- 获取任务列表失败。
- 创建任务失败。
- 更新任务状态失败。
- 删除任务失败。

### 展示要求

- 提示用户操作失败。
- 获取列表失败时，需要提供重新加载入口。
- 单条任务操作失败时，不影响其他任务展示。

## 11. 业务流程

## 11.1 页面初始化流程

1. 用户打开 Todo List 页面。
2. 前端展示加载状态。
3. 前端请求后端获取任务列表。
4. 请求成功后，展示任务列表。
5. 请求失败后，展示错误状态和重试入口。

---

## 11.2 新增任务流程

1. 用户输入任务标题。
2. 用户点击添加。
3. 前端校验标题是否为空。
4. 校验失败，提示用户输入任务内容。
5. 校验成功，请求后端创建任务。
6. 创建成功，清空输入框，并更新任务列表。
7. 创建失败，提示用户创建失败。

---

## 11.3 切换任务状态流程

1. 用户点击任务完成状态控件。
2. 前端请求后端更新任务状态。
3. 更新成功，页面展示新状态。
4. 更新失败，恢复原状态，并提示用户操作失败。

---

## 11.4 删除任务流程

1. 用户点击删除按钮。
2. 系统展示删除确认态。
3. 用户点击取消，关闭确认态，不删除任务。
4. 用户点击删除，前端请求后端删除任务。
5. 删除成功，任务从列表中移除。
6. 删除失败，任务保留，并提示用户删除失败。

---

## 11.5 筛选任务流程

1. 用户点击筛选项。
2. 前端更新当前筛选状态。
3. 前端根据任务完成状态过滤列表。
4. 页面展示筛选后的任务。
5. 如果筛选结果为空，展示对应空状态。

---

## 12. 移动端体验要求

- 页面仅需适配手机端。
- 页面宽度以移动端视口为主，不要求桌面端复杂布局。
- 按钮、输入框、列表项需要适合手指点击。
- 关键操作区域高度不应过小。
- 删除按钮需要避免误触。
- 输入任务后，应方便用户快速提交。
- 页面内容较多时，任务列表可以纵向滚动。
- 不要求复杂动画，但状态变化应清晰自然。

## 13. 验收标准

## 13.1 页面初始化

- 用户打开页面后，系统会请求后端任务数据。
- 请求过程中展示加载状态。
- 请求成功后展示任务列表。
- 请求失败后展示错误提示和重试入口。

## 13.2 新增任务

- 输入有效任务标题后，可以成功创建任务。
- 新增任务默认状态为未完成。
- 新增成功后，任务出现在当前列表中。
- 新增成功后，输入框被清空。
- 输入空内容时，不允许创建任务。

## 13.3 完成状态切换

- 未完成任务可以切换为已完成。
- 已完成任务可以切换为未完成。
- 状态切换成功后，页面展示正确状态。
- 状态切换失败时，页面提示失败，并保持合理状态。

## 13.4 删除任务

- 用户可以删除指定任务。
- 点击删除按钮后，会先展示删除确认态。
- 用户取消删除时，任务仍保留在列表中。
- 删除成功后，任务从列表中消失。
- 删除失败时，任务仍然展示在列表中，并提示失败。

## 13.5 任务筛选

- 默认展示全部任务。
- 点击「未完成」后，仅展示未完成任务。
- 点击「已完成」后，仅展示已完成任务。
- 点击「全部」后，展示所有任务。
- 当前筛选项有明确选中状态。
- 筛选结果为空时，展示对应空状态。

## 13.6 移动端适配

- 页面在常见手机屏幕宽度下可正常使用。
- 输入框、按钮、列表项无明显遮挡或错位。
- 用户可以通过触控完成所有核心操作。
- 页面不依赖桌面端鼠标悬停能力。
